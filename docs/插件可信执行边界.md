# 插件可信执行边界（部署说明）

## 默认行为

Rhex 的插件服务端入口会被动态导入；在 Node.js 中，导入模块会执行其顶层代码。因此，Rhex 默认**不执行任何插件服务端代码**，直到部署负责人完成一次明确的可信代码确认。

未设置确认变量时：

- 主站仍可正常启动；没有插件的站点不会因为缺少此变量而启动失败。
- 已禁用的插件不会被导入或执行。
- 已启用的插件会被记录为加载失败，但不会阻止其余站点功能启动；其服务端模块不会被导入或执行。
- 插件 ZIP 可以选择“安装但保持禁用”。这种安装不会评估插件的生命周期模块或其模块顶层代码。

## 允许执行可信插件

只在已经审核插件来源、压缩包内容和版本后，由**服务部署负责人**在部署环境中设置下面的精确值，并重启所有运行 Rhex 的 Web/worker 进程：

```dotenv
RHEX_ADDON_TRUSTED_CODE_ACKNOWLEDGED=I_UNDERSTAND_ADDONS_RUN_WITH_FULL_SERVER_PRIVILEGES
```

这不是普通的功能开关。该字符串刻意较长，以避免通过一个宽泛的布尔变量误开启插件执行。变量值必须完全匹配；`true`、`1`、空值或其他值都不会允许执行。

推荐启用步骤：

1. 审核插件 ZIP 的来源、代码、依赖和版本，必要时记录文件哈希。
2. 确认该插件应在与 Rhex 相同的服务端权限下运行。
3. 在部署密钥/环境配置中设置上述精确变量并重启服务。
4. 安装或启用插件。
5. 在插件生命周期日志中检查 `TRUST_BOUNDARY`（确认执行）以及 `SDK_FETCH_DENIED`（未声明网络能力的 `globalThis.fetch` 已被拒绝）事件。

如需恢复默认的“不执行插件”状态，移除该变量或将其改为任何非精确匹配的值，然后重启服务并禁用已安装插件。

## permissions 的真实含义

`addon.json` 的 `permissions` 只用于约束 **Rhex 提供的插件 API** 和记录能力申请；它不是安全沙箱，也不会缩减已导入 Node.js 代码的操作系统权限。

- 不支持、未知或声称 `sandbox:*`、`isolation:*`、`process:*` 等隔离效果的权限声明会被拒绝，插件不会执行。
- `network:external` 目前会限制插件执行范围内经由 `globalThis.fetch` 发起的请求；未声明时该路径会拒绝并记录 `SDK_FETCH_DENIED`。
- 这个 `fetch` 检查只覆盖 `globalThis.fetch`，而不是所有 Node 网络 API。

## 不可跨越的边界

已确认并导入的插件与 Rhex 位于**同一个 Node.js 进程**。恶意或不可信代码仍可能直接导入 Node 原生模块、访问进程环境、读取宿主机可访问的文件，或使用其他网络客户端绕过 Rhex SDK 的能力检查。因此：

> 当前模型是“部署负责人显式确认后执行完全可信的服务端扩展”，**不是**“可安全运行任意第三方插件的沙箱”。

不要把 Node `vm`、manifest permissions 或上述 `fetch` 检查宣传为不可信代码隔离。若业务需要运行不可信第三方扩展，应采用架构级隔离：独立进程/容器或专用 worker、受限的 IPC 协议、最小化凭据/能力令牌、资源限制、签名与人工审核，并将数据库和基础设施凭据与主站隔离。
